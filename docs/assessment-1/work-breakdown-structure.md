# Work Breakdown Structure (WBS)

## 1.0 Project Management
- 1.1 Team roles and responsibilities confirmed
- 1.2 GitHub repository and project board set up
- 1.3 Weekly meeting minutes maintained

## 2.0 Business Analysis (Isuru)
- 2.1 Stakeholder analysis
- 2.2 Business case
- 2.3 Scope definition
- 2.4 Work Breakdown Structure
- 2.5 Gantt chart
- 2.6 Risk register
- 2.7 Quality plan

## 3.0 Requirements & Design (Dineli)
- 3.1 Functional requirements
- 3.2 Non-functional requirements
- 3.3 User stories
- 3.4 UML diagrams (use case, class-level where relevant)
- 3.5 BPMN process diagrams (current-state and future-state)

## 4.0 System Architecture & AI Design (Farhaz)
- 4.1 High-level system architecture
- 4.2 Technology stack decisions
- 4.3 RAG pipeline design
- 4.4 LLM provider and vector store evaluation

## 5.0 Implementation (Farhaz, Assessment 2 onward)
- 5.1 Backend API scaffold
- 5.2 Database schema and migrations
- 5.3 RAG pipeline implementation (MVP, single provider)
- 5.4 Frontend chat interface
- 5.5 Sample policy document ingestion

## 6.0 Testing & Evaluation (Dineli writes plan / Isuru coordinates execution, Assessment 3)
- 6.1 Test case design
- 6.2 Unit and integration testing
- 6.3 UAT coordination and execution
- 6.4 AI response evaluation (groundedness, relevance)

## 7.0 Reporting & Presentation (Dineli leads, all contribute)
- 7.1 Assessment 1 report compilation
- 7.2 Final report compilation
- 7.3 Presentation slides and speaking script

**Structure note:** the WBS is organised by *deliverable stream* (Project Management, Business Analysis, Requirements & Design, Architecture, Implementation, Testing, Reporting) rather than by week, so it can be mapped directly onto the Gantt chart's timeline while staying stable if specific dates shift.
